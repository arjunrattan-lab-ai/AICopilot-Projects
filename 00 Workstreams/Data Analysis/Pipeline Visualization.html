Safety AI Event Pipeline

Edge → FIS → EFS → Annotations → Backend → FM Dashboard

Pipeline Stages
📹
Stage 1 · Edge

Dashcam (DC-53/DC-54)

Perception Engine runs UDM model on every frame. Analysis Engine watches confidence scores — if above threshold for min_duration, triggers an alert and bundles a DPE JSON composite event.

On-device
📤
Stage 2 · FIS

File Ingestion Service

Receives DPE JSON from edge via S3. Stores compressed .json.gz in keep-truckin-data-lake. Passes event to EFS for filtering.

Cloud
🔍
Stage 3 · EFS

Event Filtering Service

Post-filters for false positives: speed mode, low FPS, obstruction, face proximity. Sets video_required and annotations_required flags. Alerts are NOT processed here — only logged.

Cloud
🏷️
Stage 4 · Annotations

Human Review Tool

Annotators review video clips and apply behavioral tags per product definitions. True positives flow to backend; false positives feed model retraining.

Manual Review
🖥️
Stage 5 · Backend

K2Web Backend

Applies capping logic per driver/period. Records events in DRIVER_PERFORMANCE_EVENTS with coaching status, risk level, driver ID (face_match), and location data.

K2Web
🔊
Parallel · In-Cab Alerts

Alerts Logging

Only played alerts (audio heard in-cab) are recorded in IN_CAB_ALERTS_WH. Silenced alerts (backoff/cooldown/mode=1) are still logged to event tables but not here.

⚠️ Key Gotcha

The ALERT_STATUS field in EFS event logs does NOT confirm the alert was played. It only means an alert was associated with the event — it can still be silenced by backoff, cooldown, or mode=1. The only definitive proof is the SAFETY.IN_CAB_ALERTS_WH table. Use created_at timestamps (not IDs) to correlate back to DPE events.
