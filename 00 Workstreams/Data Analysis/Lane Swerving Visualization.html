Lateral Displacement (LD)

A single continuous value 0 → 1 representing where the vehicle center sits between its two detected lane lines at any given frame.

0.0 = center on right lane line  ·  0.5 = dead center  ·  1.0 = center on left lane line

Everything downstream — swerve detection, lane change exclusion, drift filtering — is purely pattern-matching on the shape of this signal over time. The perception engine outputs it; the state machine interprets it.

When a lane change occurs, the perception engine switches from tracking the old lane lines to the new ones, causing an apparent jump of ~1.0 in LD. That discontinuity is the lane change signal.

Signal Shapes — What Each Pattern Looks Like
True Swerve Correct Detection Target
Hill or valley oscillation — car moves laterally then returns
Shape: Clear peak or trough. Start, peak, end form a hill/valley. Both sides drop ≥0.15 from peak; one side ≥0.30. Duration: typically 3–8s. Amplitude: 0.15–0.30 of lane width.
Drift V1 Bug — False Positive
Monotonic slope — car slowly slides in one direction without returning
Shape: Gradual one-way slide. No true peak — one "side" is always higher than the other. V1 can find 3 zero-velocity points on the slope and misclassify as swerve. V2 dual-threshold rejects it: you can't have both sides ≥0.15 below a peak on a monotonic slope.
Lane Change Filtered Out
Step function — LD jumps ~1.0 as perception engine switches lane lines
Shape: Near-instant jump of ~1.0 in LD (old lane → new lane tracking). V1: explicit Lane Change Exclusion Filter drops event if jump detected mid-swerve. V2: numpy.unwrap-style correction subtracts the jump; also flags any displacement >0.65 as lane change.
Boomerang Lane Change Gray Zone — Top FP Source
Overtake-and-return — looks swerve-shaped but at lane-change amplitude (~0.5+)
Shape: Hill/valley shape but wide — amplitude 0.5+ (above swerve, below full lane change threshold of 0.65). Falls in the gray zone between 0.30 and 0.65. Identified as top FP source in the June 2025 tech review. Neither V1 nor V2 reliably filters these.

What triggers the fire: Amplitude (peak vs. start/end difference) must exceed threshold AND duration of the excursion must be within the valid window. This is where V1 and V2 critically diverge — V1 does not verify the peak is a true extremum; V2 enforces strict dual-threshold shape checking.
V2 Dual-Threshold Shape Check

V2 requires the candidate peak to be a true extremum with both surrounding points sufficiently below it. A monotonic drift cannot satisfy this — one side will always be above the "peak."

V1 (Edge) vs. V2 (Cloud) — Key Differences

V1 — Edge (DC-53/DC-54)

The current on-device implementation — has the drift bug

No extremum enforcement: finds 3 zero-velocity points but doesn't verify the middle one is a true peak. Can pick 3 points on a monotonic slope.

Lane Change Exclusion Filter: drops event if a lane change is detected mid-swerve. Asad Soomro requested: "drop event and move to None state."

No drift filter: slow, gradual lateral slides pass through as swerve events — this is the current bug on VG5.

V2 — EFS / Cloud

The correct post-processing implementation

Exhaustive search with shape constraints: peak must be a true max or min. Start: 2–10s before peak. End: 2–10s after peak.

Dual amplitude threshold: both start & end must be ≥0.15 below peak; at least one must be ≥0.30 below. Drift physically cannot pass this.

Lane width consistency check: lane width must stay between 0.83× and 1.2× of width at peak throughout the window.

numpy.unwrap-style LC correction: when LD jumps ~1.0, adjusts signal to be continuous so downstream pipeline isn't fooled. Also flags anything >0.65 as lane change.

2100ms lookback window for reference points in LC detection.
